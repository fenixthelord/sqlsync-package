# SqlSync — دروس ومشاكل شائعة (Troubleshooting)

توثيق شامل للمشاكل الحقيقية التي واجهناها أثناء بناء وتشغيل نظام SqlSync (البرنامج على الويندوز، باكج Laravel، إضافة Filament)، وحلولها. الهدف: أي حد (بما فيه أنا بعد شهر) يقرأ هاد الملف قبل ما يعيد اكتشاف نفس المشكلة من الصفر.

---

## 1. برنامج الويندوز (.NET Agent)

### 1.1 حقول Settings بتطلع فاضية كل ما تفتح الشاشة، وبتمسح القيم المحفوظة
**السبب:** الكود كان بيبني الحقول (TextBox) من جديد كل مرة تفتح تبويب، بدون كود يحمّل القيمة المحفوظة من `ConfigService`. النتيجة: تفتح الإعدادات، تشوف الحقل فاضي (أو بس Placeholder رمادي زي "COMP001")، تنسى تعيد كتابة القيمة، تضغط Save — وبتنمسح القيمة الحقيقية بسترينغ فاضي.

**الحل:** أضف دالة `PopulateXxxFields()` تُستدعى فوراً بعد بناء كل تبويب، تقرأ من `_configService.Config` وتعبّي كل TextBox/CheckBox قبل ما المستخدم يشوف الشاشة.

**درس عام:** أي شاشة إعدادات بتبنى ديناميكياً، تأكد فيه مسار "تحميل" مقابل لكل مسار "حفظ".

### 1.2 أزرار Radio (Provider: Auto/Al-Ameen/Al-Bayan/Manual) ما بتلغي بعض
**السبب:** كل زر Radio كان جوا Panel منفصل (تصميم بصري كروت). WinForms بيربط الـ RadioButtons ببعض تلقائياً **بس** إذا كانوا بنفس الأب المباشر (parent container). بما إنو كل واحد بصندوقه الخاص، اختيار "Al-Bayan" ما كان يلغي "Al-Ameen" تلقائياً — الاثنين يضلوا `Checked = true`.

**الحل:** اربط يدوياً حدث `CheckedChanged` لكل الأزرار الأربعة، وعند تفعيل أي واحد، ألغِ الباقي صراحة بالكود.

**درس عام:** أي مجموعة أزرار Radio موزّعة على Containers منفصلة (لأسباب تصميم/تنسيق) تحتاج ربط يدوي إجباري.

### 1.3 أيقونة جديدة عند الساعة (Tray) كل ما تحفظ الإعدادات
**السبب:** دالة `BuildTrayIcon()` كانت بتعمل `new NotifyIcon` من الصفر بدون Dispose للقديمة، فكل استدعاء بيضيف أيقونة جديدة بدل ما يستبدل القديمة.

**الحل:** بداية الدالة، تحقق إذا فيه `_trayIcon` موجود مسبقاً، خليه `Visible = false` وسوّي `Dispose()` له قبل إنشاء الجديد.

### 1.4 مزامنة عدد ضخم من الأصناف (20 ألف+) بتعمل Timeout
**السبب:** إرسال كل السجلات بطلب HTTP واحد ضخم.

**الحل جزئين:**
- **Batching:** قسّم الإرسال لدفعات (500 بالدفعة افتراضياً)، أرسلهم بالتسلسل.
- **Incremental Sync:** لو المصدر عندو عمود موثوق "آخر تعديل" (مثل `UDate` بجدول `MatCard` بالبيان)، احفظ آخر توقيت مزامنة ناجحة، وبالدورة الجاية اسأل بس عن `WHERE UDate > آخر_توقيت` بدل إعادة سحب كل الجدول من الصفر.

## 2. الباكج الأساسي (sqlsync/laravel-sqlsync) والـ Product Bridge

### 2.1 كل عملية Sync بترجع أسعار = 0 أو فئات فاضية
**السبب:** الـ Preset (`AlAmeenPreset.php` / `AlBayanPreset.php`) كان مبني على **افتراض** أسماء أعمدة البيانات الخام (`$raw['group']`, `$raw['mtRetail']`)، بدل التأكد من الاسم الحقيقي يلي بيرجع فعلياً من الـ Windows Agent.

**الحل:** قبل ما تكتب أي Preset، خلي الـ Agent يرسل عينة حقيقية أول، وابني الـ mapping بناءً على أسماء الحقول الفعلية بالضبط — لا تخمّن.

### 2.2 `AlBayan` أسعارها (Price1-35) بلا معنى ثابت
**المشكلة:** برامج زي البيان عندها أعمدة عامة (`Price1` لـ `Price35`) وكل شركة عم تحدد معناها بإعدادات برنامجها الخاص — نفس العمود ممكن يكون سعر جملة عند شركة وسعر تجزئة عند شركة تانية.

**الحل:** لا تحاول تخمّن أو تثبّت عمود واحد بالكود. خزّن كل الأعمدة كما هي بـ `extra_data`، واخلي "Product Bridge" (إعداد مرئي بالفيلامنت لكل شركة) يحدد أي عمود يعني شو.

### 2.3 التصنيف (Category) قيمه غريبة بدون جدول لوكأب واضح
**الدرس:** لا تفترض إنو أي عمود اسمه "Class" أو "Group" هو فعلياً تصنيف عام. افحص القيم الفعلية (لو عندك 200+ قيمة مختلفة توزيعها غريب، افحص إذا فعلياً Brand/Supplier مش Category). لو ما قدرت تتأكد من داخل قاعدة البيانات، اطلب فحص الشاشة الفعلية ببرنامج المحاسبة نفسه — مو تخمين من أسماء الأعمدة.

### 2.4 منتج جديد بدون فئة → فشل الإنشاء (category_id إجباري)
**الحل:** ميزة **Auto-Category** — لو الفئة القادمة من السجل مش موجودة بجدول الفئات عندك، تتخلق تلقائياً (find-or-create) وتترابط. لو ما حددت هاي الميزة، أي صنف بدون فئة يضل بانتظار مراجعة يدوية بدل ما يفشل بصمت.

### 2.5 باركود مكرر بالبيانات الحقيقية (أكتر من صنف بنفس الباركود)
**الدرس:** هاد طبيعي وشائع (أخطاء إدخال، عيّنات، أصناف قديمة). أي عملية Bulk (Sync حقيقي أو "إعادة تطبيق الربط") لازم تكون محصّنة **سجل سجل** (`try/catch` حول كل صف لحاله)، وإلا صف واحد فاشل بيوقف كل العملية.

### 2.6 حقل إجباري (مثل `price`, `name_ar`, `slug`) بيرجع فاضي أحياناً
**الحل:** غلّف عمليات `create()`/`update()` جوا الـ Observer نفسه بـ `try/catch`، سجّل السبب، وكمّل للسجل التالي. **لا تعتمد على إنو الطرف المستدعي (Controller أو زر بالواجهة) هو يلي رح يحمي العملية** — الحماية لازم تكون بأعمق نقطة ممكنة (جوا الـ Observer).

### 2.7 نسخة قديمة من كود مكرر بمكانين بنفس الوقت
**أخطر مشكلة بالجلسة:** نقلنا `SyncedRecordBridgeObserver` من مستوى المشروع (`app/Observers/`) لمستوى الباكج (`vendor/sqlsync/...`) لجعله تلقائي التسجيل — بس ما حذفنا النسخة القديمة من المشروع ولا ألغينا تسجيلها. النتيجة: **نسختين تشتغلوا بنفس الوقت** على نفس الحدث (`saved`) — وحدة محمية (بالباكج) ووحدة بدون حماية (بالمشروع القديم)، وكانت القديمة هي يلي بتنهار وتوقف كل العملية، بينما كل الأدلة (فحص كود الباكج، composer show، Opcache) كانت بتقول "كل شي سليم".

**الدرس الأهم بالجلسة كلها:** لما تنقل كود من مستوى لمستوى (مشروع → باكج، أو أي نقل مشابه)، **لازم تتأكد تحذف النسخة القديمة بالكامل** وتلغي أي تسجيل يدوي إلها. وإذا واجهت سلوك متناقض (الكود شكله صحيح بس النتيجة غلط)، افحص **كل الأماكن يلي ممكن يكون فيها نفس الكلاس/المنطق مسجّل**، مش بس المكان "الرسمي".

## 3. الاستضافة المشتركة (cPanel / Shared Hosting)

### 3.1 عدّلت الكود، رفعته، بس السلوك ما تغيّر
**غالباً Opcache.** تحقق:
```php
opcache.validate_timestamps   // لو 1، بيتحقق تلقائياً كل revalidate_freq ثانية
opcache.revalidate_freq       // كل كم ثانية يتحقق
```
لو `validate_timestamps=0`، لازم إعادة تشغيل PHP-FPM يدوياً (أو `opcache_reset()` عبر Route مؤقت بالويب، **ليس CLI/Tinker** — الاثنين عمليات PHP منفصلة بذاكرة Opcache منفصلة أحياناً).

**تنبيه:** لا تفترض دايماً إنو Opcache هو السبب. جرب `grep "try {"` مباشرة على ملف الـ vendor عالسيرفر للتأكد القاطع قبل ما تضيع وقت بإعادة تشغيل أشياء.

### 3.2 عملية طويلة (18000+ سجل) بتعمل "Maximum execution time exceeded"
**الحل الصح:** ماتزيدش `max_execution_time` — استخدم **Queue** (`QUEUE_CONNECTION=database`) + **Cron Job** (كل دقيقة: `php artisan queue:work --stop-when-empty --max-time=55`). المستخدم النهائي بيضغط زر بالواجهة بس، والعملية تشتغل بالخلفية بدون حد زمني.

**لا تستخدم حل Terminal (`php artisan command`) كحل نهائي** إذا في مستخدمين نهائيين غير مطورين — الـ Terminal للمطور بس وقت التطوير/التشخيص.

### 3.3 جدول `jobs` مش موجود لما فعّلت Queue لأول مرة
```bash
php artisan queue:table
php artisan migrate
```

## 4. Laravel Pint / CI

- **`fully_qualified_strict_types`**: لا تستخدم `\Namespace\Class::method()` جوا الكود — دايماً `use` بالأعلى واستخدم الاسم المختصر.
- **`ordered_imports`**: كل أسطر `use` لازم تكون مرتبة أبجدياً.
- **دالة فاضية تماماً** (`function boot(): void {}`) ممكن تسبب تعارض قواعد (`braces_position` vs `single_line_empty_body`). الحل الآمن: استخدم نفس قالب Laravel الافتراضي (`{ // }` بتعليق فاضي جوا).
- **نهايات الأسطر (CRLF vs LF):** لو حررت ملف على ويندوز أو نسخته بطريقة غير مباشرة، تأكد إنو بصيغة LF قبل الـ commit.

## 5. Filament (واجهة الإدارة)

- **View مخصص (Blade مكتوب يدوياً) ما بيعرض `getFormActions()` تلقائياً** — إذا بنيت صفحة Filament بـ View خاص، أي زر إضافي لازم تحطه بالـ View نفسه صراحة، مش بس بالـ PHP.
- **أسماء Slots غير موثّقة** (زي `headerEnd`) ممكن ما تكون مدعومة بكل نسخة Filament — لا تخمّن اسم Slot، جرب الـ Slots الأساسية المعروفة (`heading`, `description`) وحط أي محتوى إضافي بالـ body العادي إذا مش متأكد.
- **كلاسات Tailwind يدوية بملف Blade ضمن باكج خارجي** ممكن ما تكون موجودة بالـ CSS المبني للتطبيق المضيف. للتباعد الحرج، استخدم `style="margin-top: ..."` مباشر بدل الاعتماد على كلاسات Tailwind غير مضمونة.

## 6. أمان

- **لا تشارك GitHub Personal Access Token بنص صريح بأي محادثة أو قناة غير آمنة** — أي حد يشوف المحتوى بعدها بياخد نفس الصلاحيات. إذا حصل، ألغِ التوكن فوراً من `github.com/settings/tokens` وولّد وحدة جديدة بصلاحيات محدودة (repo واحد بس، مش كل الحساب).
